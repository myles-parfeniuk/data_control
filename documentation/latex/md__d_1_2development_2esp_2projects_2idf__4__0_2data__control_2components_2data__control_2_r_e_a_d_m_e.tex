\doxylink{namespace_data_control}{Data\+Control} is a C++ component written for esp-\/idf version 4.\+0, intended to simplify the management of data and actions associated with that data.

{\bfseries{What does it do?}}
\begin{DoxyItemize}
\item Allows the creation of Data\+Wrapper objects implemented as a template class, used to store data of any type.
\item Call-\/back functions can be registered to Data\+Wrapper objects.
\item Call-\/back functions will be executed when the data of Data\+Wrapper object is modified.
\end{DoxyItemize}

{\bfseries{Why would I want to use it?}}

In embedded system applications it\textquotesingle{}s often required to take some sort of action that is associated with a piece of data. ~\newline
 For example, we might want to\+:


\begin{DoxyItemize}
\item Turn on an LED and power on a peripheral when a push-\/button is pressed
\item Adjust the PWM of a fan and write its speed to an OLED or LCD after sampling its tach signal
\item Write the distance measured by a sensor to an OLED or LCD and move a stepper motor, after taking a distance reading
\end{DoxyItemize}

Firmware can become large and complicated due to actions like these, especially in the context of a large system where several actions may need to be associated with one piece of data. \doxylink{namespace_data_control}{Data\+Control} is intended to remedy this issue.

Now, in the context of the above examples, with \doxylink{namespace_data_control}{Data\+Control} we can\+:


\begin{DoxyItemize}
\item Store the status of the button in a Data\+Wrapper object within a \"{}\+Button\"{} class, register 2 different call-\/backs to Data\+Wrapper in separate classes (for ex. \"{}\+Led\+Backend\"{} \& \"{}\+My\+Peripheral\"{})
\item Store the speed of the fan in a Data\+Wrapper object within a \"{}\+Fan\"{} class, register 2 different call-\/backs to Data\+Wrapper in separate classes (for ex. \"{}\+Oled\+Backend\"{} \& \"{}\+PWMBackend\"{})
\item Store the distance measured by the sensor in a Data\+Wrapper object within a \"{}\+Distance\+Sensor\"{} class, register 2 different call-\/backs to Data\+Wrapper in separate classes (for ex. \"{}\+Oled\+Backend\"{} \& \"{}\+Stepper\+Backend\"{})
\end{DoxyItemize}

This leads to well organized firmware that is much easier to work with, especially as a system expands and grows.

It was originally inspired by an in-\/house component I used at a workplace; however, this component has been re-\/written without using that component as a reference. ~\newline

To instantiate a Data\+Wrapper, instantiate one of its sub-\/classes. The differences are as follows\+:


\begin{DoxyItemize}
\item Call\+Always will execute follower callback functions {\itshape always}
\item Call\+Different will only execute follower callback functions if the current data is {\itshape different} from the new data set()
\item Call\+Same will only execute follower callback functions if the current data is the {\itshape same} as the new data set()
\item Call\+Equals will only execute follower callback functions if the new data set() {\itshape equals} its compare\+\_\+data member ~\newline


Example syntax\+: ~\newline

\end{DoxyItemize}


\begin{DoxyCode}{0}
